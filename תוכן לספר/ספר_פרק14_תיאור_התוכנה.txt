14.1  שפות תכנות
------------------------------------------------------------------------

הפרויקט פותח תוך שימוש במספר שפות תכנות וטכנולוגיות, כל אחת
מהן נבחרה בהתאם לשכבה שבה היא פועלת ולדרישות הספציפיות של
אותה שכבה.


שפת C# — צד השרת

C# היא שפת התכנות הראשית של הפרויקט. כל לוגיקת הצד השרת,
כולל האלגוריתמים, בדיקות היחידה ושכבת ה-API, מומשה ב-C#.

סיבות לבחירה:
  — ביצועים גבוהים לחישובים אינטנסיביים כגון Branch & Bound.
  — תמיכה מלאה ב-OOP ועיצוב מודולרי לפי עקרונות SOLID.
  — אינטגרציה טבעית עם ASP.NET Core ו-Entity Framework Core.
  — טיפוסיות חזקה (Strong Typing) המפחיתה שגיאות זמן ריצה.
  — תמיכה ב-Async/Await לטיפול יעיל בבקשות HTTP מקביליות.

השפה שימשה לכתיבת מגוון רכיבים:
  • מודלים של Domain (Box, Container, PackingResult ועוד).
  • שלושת האלגוריתמים: BranchAndBoundSolver, GreedySolver,
    HeuristicH1.
  • Controllers של ה-API המקבלים ומחזירים JSON.
  • שכבת הנתונים: EF Core DbContext, Repository.
  • בדיקות יחידה עם xUnit.


JavaScript — צד הלקוח

JavaScript שימשה לכתיבת כל לוגיקת הממשק. כל דפי ה-HTML
מכילים קוד JavaScript המבצע קריאות לשרת, עיבוד נתוני התשובה
והצגתם למשתמש.

שימושים עיקריים:
  — שליחת בקשות HTTP לשרת (Fetch API) עם Authorization
    header הנושא את ה-JWT Token.
  — שמירה וניהול ה-Token ב-sessionStorage.
  — בניית ממשק הטופס לבחירת ארגזים וקונטיינרים.
  — הצגת תוצאות השיבוץ: אחוז ניצול, רשימת ארגזים, הודעות שגיאה.

ספריית Three.js שימשה לצד JavaScript לרינדור התצוגה
התלת-ממדית האינטראקטיבית. בעזרתה הוצגה כל תוצאת שיבוץ
כמודל 3D שניתן לסובב ולבחון ממגוון זוויות.


HTML ו-CSS

HTML5 שימשה לבניית מבנה דפי הממשק. הפרויקט כולל שישה
דפי HTML עצמאיים הנפרשים ישירות בדפדפן ללא צורך בסביבת
ריצה נוספת.

CSS3 שימשה לעיצוב האחיד של כל הממשק: פריסה, צבעים,
כפתורים, טבלאות ותצוגת המידע הסטטיסטי.


SQL — שפת מסד הנתונים

סקריפטים בשפת SQL שימשו ליצירת הטבלאות ולמילוי נתוני
ההתחלה (Seed Data). הגישה ממסד הנתונים בזמן הריצה מתבצעת
באמצעות Entity Framework Core, ללא כתיבת SQL ישיר.


14.2  סביבת העבודה
------------------------------------------------------------------------

הפרויקט פותח בסביבת Windows תוך שימוש בכלים מקצועיים
המקובלים בתעשיית הפיתוח.


סביבת פיתוח — Visual Studio

Visual Studio 2022 שימש כסביבת הפיתוח המרכזית (IDE).
הסביבה כוללת:
  — עורך קוד עם תמיכה מלאה ב-C# ו-IntelliSense.
  — ניהול ה-Solution הכולל את כל פרויקטי ה-.NET.
  — מנפה שגיאות (Debugger) אינטגרטיבי להרצת קוד
    שלב אחר שלב.
  — חלון מובנה להרצת בדיקות יחידה (Test Explorer).
  — ניהול חבילות NuGet ישירות מתוך הסביבה.

שאר קבצי הפרויקט (HTML, CSS, JavaScript) נערכו
ב-Visual Studio Code שהוא עורך קל-משקל ומהיר המתאים
לעבודה בצד הלקוח.


סביבת ריצה — .NET 10 SDK

כל שכבת השרת מבוססת על .NET 10 SDK. סביבה זו מספקת:
  — ניהול יחסי התלות בין הפרויקטים השונים.
  — הידור ובניית הקוד (Build).
  — הרצת השרת דרך שורת הפקודה או מתוך Visual Studio.
  — הפעלת בדיקות היחידה (dotnet test).


מסד נתונים — SQL Server LocalDB

מסד הנתונים מנוהל באמצעות SQL Server LocalDB,
גרסה קלת-משקל המגיעה יחד עם Visual Studio ומתאימה
לסביבת פיתוח מקומית.

גישה ממסד הנתונים מתבצעת דרך Entity Framework Core
(EF Core) — ORM המאפשר עבודה עם טבלאות SQL בממשק
אובייקטים של C# מבלי לכתוב שאילתות SQL ישירות.

התחברות לבסיס הנתונים מוגדרת ב-appsettings.json
ומוזנת לשרת בהפעלה.


בקרת גרסאות — Git

ניהול הגרסאות בוצע באמצעות Git. מאגר הקוד מוחזק
ב-GitHub ומאפשר:
  — מעקב מלא אחר שינויים לאורך כל שלבי הפיתוח.
  — יכולת חזרה לגרסה קודמת בעת הצורך.
  — גיבוי שוטף של כל קוד המקור.


14.3  פירוט ה-API
------------------------------------------------------------------------

שרת OptiLoad חושף ממשק API מסוג REST. כל בקשה מחזירה
תשובה בפורמט JSON. נקודות הקצה מחולקות לקבוצות לפי
הפונקציה שהן מבצעות.

כל הנקודות (למעט אימות) מוגנות ב-JWT Authentication —
בקשה ללא Token תקף תדחה עם קוד תגובה 401.


אימות משתמשים — AuthController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  POST  /api/auth/login
    קלט:  { "username": "...", "password": "..." }
    פלט:  { "token": "<JWT>", "expiresIn": 7200 }
    תיאור: מקבל שם משתמש וסיסמה, מאמת מול מסד הנתונים
           ומחזיר JWT Token בתוקף לשעתיים.
           לאחר שלושה ניסיונות כושלים מאותו IP, הגישה
           נחסמת למשך חמש דקות (Rate Limiting).


ניהול ארגזים — BoxController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/boxes
    פלט:  רשימת כל הארגזים השמורים במסד הנתונים.
    תיאור: מחזיר מערך של אובייקטי Box הכולל שם, מידות,
           משקל, סטטוס שבירות ואפשרות סיבוב לכל ארגז.

  POST  /api/boxes
    קלט:  אובייקט Box חדש.
    פלט:  הארגז שנוצר עם ה-Id שהוקצה לו.

  PUT   /api/boxes/{id}
    קלט:  אובייקט Box מעודכן.
    פלט:  204 No Content בהצלחה.

  DELETE /api/boxes/{id}
    פלט:  204 No Content בהצלחה.


ניהול קונטיינרים — ContainerController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/containers
    פלט:  רשימת כל הקונטיינרים השמורים.
    תיאור: כולל מידות (Width, Height, Depth) ומגבלת
           משקל (MaxWeightKg) לכל קונטיינר.

  POST  /api/containers
  PUT   /api/containers/{id}
  DELETE /api/containers/{id}
    ניהול מלא (CRUD) של קונטיינרים.


הרצת האלגוריתם ותצוגה — VisualizationController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  POST  /api/visualization/run
    קלט:  {
            "container": { "width": ..., "height": ...,
                           "depth": ..., "maxWeightKg": ... },
            "boxes":     [ { "name": "...", "w": ..., "h": ...,
                             "d": ..., "weight": ...,
                             "fragile": ..., "allowRotation": ...,
                             "qty": ... } ],
            "timeLimitSeconds": 10,
            "loadingFace": "Front"
          }
    פלט:  אובייקט PackingResult הכולל:
            — binsUsed          : מספר הקונטיינרים שנדרשו
            — volumeUtilization : אחוז ניצול הנפח
            — isOptimal         : האם הפתרון מובטח אופטימלי
            — statusMessage     : הודעת סטטוס מילולית
            — solveTime         : זמן החישוב
            — placedBoxes       : רשימת מיקומי הארגזים בתלת-ממד
            — unplacedBoxes     : ארגזים שלא ניתן היה לשבץ
            — perBinStats       : סטטיסטיקות לכל קונטיינר
    תיאור: זוהי נקודת הקצה המרכזית של המערכת. היא מפעילה
           את אלגוריתם ה-Branch & Bound עם ה-Heuristic
           כנקודת פתיחה, ואת ה-Greedy כ-Fallback.


ניהול עבודות שיבוץ — PackingJobController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/packingjobs
    פלט:  רשימת כל עבודות השיבוץ של המשתמש המחובר.

  POST  /api/packingjobs
    קלט:  פרטי עבודת שיבוץ חדשה.
    פלט:  העבודה שנוצרה.
    תיאור: שומרת הגדרות ריצה לשמישות עתידית ולמעקב.

  GET   /api/packingjobs/{id}
    פלט:  פרטי עבודה ספציפית לפי מזהה.


ניהול תמונות מצב — SnapshotController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/snapshots
    פלט:  רשימת כל ה-Snapshots של המשתמש המחובר.

  POST  /api/snapshots
    קלט:  תוצאת שיבוץ לשמירה.
    פלט:  ה-Snapshot שנוצר עם חותמת זמן.
    תיאור: מאפשר לשמור תוצאת ריצה ולהשוות אותה בהמשך
           לריצות אחרות.

  GET   /api/snapshots/{id}
  DELETE /api/snapshots/{id}
    שליפה ומחיקה של Snapshot ספציפי.


ניהול Sessions — SessionController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/sessions
  POST  /api/sessions
  DELETE /api/sessions/{id}
    ניהול Sessions של עבודה. Session מאפשר קיבוץ מספר
    עבודות שיבוץ תחת מטריה אחת לצורך ניתוח והשוואה.


ייצוא — ExportController
- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -

  GET   /api/export/{snapshotId}
    פלט:  קובץ JSON להורדה הכולל תכנית טעינה מסודרת.
    תיאור: מייצר קובץ עם סדר הטעינה המחושב לפי כיוון
           הדלת (Loading Face), מוכן לשימוש במערכות
           ניהול מחסן חיצוניות.
