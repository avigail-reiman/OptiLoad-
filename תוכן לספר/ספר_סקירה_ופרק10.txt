10.4 ביצועים עיקריים
---------------------

המנוע האלגוריתמי של OptiLoad מבוסס על גישה היברידית
המשלבת שלוש שכבות:

א. Branch & Bound (BranchAndBoundSolver):
   אלגוריתם חיפוש מדויק המחפש את מספר הקונטיינרים
   המינימלי לאריזת כלל הארגזים.
   - מוגבל ל-10,000,000 צמתים ועד 300 שניות זמן ריצה
   - מפעיל קיצוץ (Pruning) על בסיס שלושה חסמים תחתונים:
     * L0 — חסם נפח טהור
     * L1 — חסם עם פיצול לאורך ציר
     * L2 — חסם מחוזק
   - מיקומי המועמדים נקבעים על ידי CornerPointsFinder
     שמחפש "נקודות פינה" חוקיות בחלל הקונטיינר

ב. HeuristicH1:
   מספקת פתרון פתיחה מהיר לפני שמתחיל החיפוש המלא.
   - מריצה שלושה ניסיונות מיון (לפי עומק / רוחב / גובה)
     ובוחרת את הפתרון בעל ניצול הנפח הגבוה ביותר
   - מצמצמת משמעותית את מרחב החיפוש של Branch & Bound

ג. Greedy Fallback (GreedySolver):
   מופעל אוטומטית כאשר Branch & Bound מגיע למגבלת
   הזמן או הצמתים.
   - מבטיח שהמערכת תמיד תחזיר פתרון גם בקלטים גדולים
   - מורכבות: O(n * corners * rotations)

תוצאות בדיקה:
   - ניצול נפח ממוצע: מעל 80% בתרחישי ארגזים בגדלים מעורבים
   - מספר בדיקות יחידה עוברות: 20/20 (xUnit)


10.5 אילוצים
-------------

אילוצים גיאומטריים:
  - כל ארגז חייב להיות ממוקם כולו בתוך גבולות הקונטיינר
    (0 ≤ X+W ≤ ContainerWidth, וכן לגובה ולעומק)
  - אין חפיפה מרחבית בין אף שני ארגזים
  - ארגז המסומן כשביר (IsFragile = true) לא יכול לשאת
    עליו ארגז אחר
  - ארגז שסיבובו אסור (AllowRotation = false) יוצב בכיוונו
    המקורי בלבד

אילוץ משקל:
  - סך משקל כלל הארגזים הממוקמים בקונטיינר לא יחרוג
    מכושר הנשיאה המוגדר (MaxWeightKg)

אילוצי תוכנה:
  - מסד הנתונים OptiLoadDB חייב להיות מותקן ונגיש לפני
    הפעלת השרת; בהיעדרו — השרת נכשל בהפעלה
  - מפתח ה-JWT בקובץ appsettings.json חייב להיות לפחות
    32 תווים; אורך קצר יותר גורם לחריגה בעת ההפעלה

אילוצי ביצועים:
  - זמן ריצת אלגוריתם שיבוץ מוגבל ל-300 שניות לכל קריאת API
  - מספר תמונות מצב בבקשה בודדת מוגבל ל-500
  - גודל תמונה בודדת (Base64) מוגבל ל-8 MB

אילוצי ממשק:
  - ממשק הלקוח פועל בדפדפן Web מודרני התומך ב-ES6 ו-WebGL
    (נדרש ל-Three.js)
  - גישה לממשק המנהל דורשת טוקן JWT תקין; ללא אימות —
    הדפדפן מנותב לדף הכניסה

11. תיאור הארכיטקטורה

11.1 הארכיטקטורה של הפתרון המוצע (Top-Down Design)

המערכת פותחה על פי עקרון ה-Clean Architecture (ארכיטקטורה נקייה), המחלק את
הפרויקט לשכבות עצמאיות עם כיוון תלות חד-סטרי — כל שכבה חיצונית תלויה
בפנימית בלבד, ולא להפך. להלן תיאור מלמעלה למטה:

שכבה 1 — ממשק המשתמש (Client / Frontend)
ממשק Web סטטי הכולל 6 דפי HTML. דרכו מגדיר המנהל ארגזים וקונטיינרים,
מריץ את האלגוריתם וצופה בתוצאות בתצוגה תלת-ממדית. מסכי השטח מאפשרים
לצוות הטעינה לצפות בתכנית ללא חשבון מנהל.

שכבה 2 — שרת האפליקציה (Backend / API)
ASP.NET Core Web API המאזין על פורט 5098. מכיל 8 בקרים האחראים לניתוב
בקשות HTTP, אימות JWT, הפעלת הלוגיקה העסקית ושמירת תוצאות במסד הנתונים.

שכבה 3 — לוגיקה עסקית (Core)
גרעין המערכת — OptiLoad.Core. מכיל את כל האלגוריתמים (B&B, Greedy, H1),
מודלי הדומיין (Box, Container, PackingJob וכד') וממשקי Repository. שכבה זו
אינה תלויה בשום רכיב חיצוני; ניתנת לבדיקה מבודדת ולהחלפת שכבות חיצוניות
ללא שינוי הגרעין.

שכבה 4 — גישה לנתונים (Data)
OptiLoad.Data מממשת את ממשקי ה-Repository באמצעות ADO.NET ישיר מול SQL Server
LocalDB. כל שאילתה מבוצעת עם Parameterized Query למניעת SQL Injection.

שכבה 5 — מסד הנתונים (Database)
SQL Server LocalDB, מסד OptiLoadDB. מאחסן מנהלים, ארגזים, קונטיינרים,
משימות שיבוץ, תוצאות מיקום, תמונות מצב וסשנים משותפים.

תרשים תלויות (מלמעלה למטה):
  Client (HTML/JS)
      ↓  HTTP/REST
  OptiLoad.API
      ↓  DI / Interfaces
  OptiLoad.Data  →  OptiLoad.Core (Domain + Algorithms)
      ↓  ADO.NET
  SQL Server LocalDB


11.2 תיאור הרכיבים בפתרון

  רכיב              | טכנולוגיה              | תיאור
  -------------------|------------------------|------------------------------------------
  שרת (Backend)      | C# .NET 10             | ASP.NET Core Web API, 8 בקרים, JWT Auth,
                     |                        | Middleware, LoginRateLimiter (IP-based)
  לוגיקה (Core)      | C# .NET 10             | אלגוריתמי B&B / Greedy / H1; מודלי דומיין;
                     |                        | 4 ממשקי Repository; PasswordHasher
  לקוח (Frontend)    | HTML / CSS / JS        | 6 דפים סטטיים, Three.js לתצוגה 3D,
                     |                        | Fetch API לתקשורת עם השרת
  מסד נתונים         | SQL Server LocalDB     | ADO.NET ישיר, Parameterized Queries,
                     |                        | מסד בשם OptiLoadDB

C# .NET 10 — שפת הפיתוח הראשית בצד השרת. .NET 10 מאפשרת ביצועים גבוהים,
ניהול זיכרון אוטומטי ותמיכה מובנית ב-async/await לטיפול בבקשות במקביל.
ה-Solution בנוי בארכיטקטורת Clean Architecture עם ארבעה פרויקטים — Core,
Data, API ו-Tests — כאשר תלויות הפרויקטים הולכות רק כלפי ה-Core.

HTML / CSS / JavaScript + Three.js — ממשק הלקוח מומש כאפליקציית Web סטטית
ללא תלות ב-Framework. Three.js משמשת לרינדור תלת-ממדי אינטראקטיבי של
תוצאות השיבוץ ישירות בדפדפן תוך שימוש ב-WebGL. Fetch API מאפשר תקשורת
אסינכרונית עם השרת.

SQL Server LocalDB — מערכת ניהול מסד נתונים רלציונית מבוססת SQL Server,
המותקנת לוקאלית ללא צורך בשרת נפרד. גישה למסד מתבצעת דרך ADO.NET עם
Parameterized Queries בלבד, ללא שימוש ב-ORM.


11.3 תיאור תהליכי מערכת ההפעלה המתבצעים בפרויקט

בזמן ריצת המערכת מתבצעים תהליכי מערכת הפעלה הבאים:

.1 תהליך שרת ה-API (dotnet.exe) — תהליך .NET המאזין על פורט TCP 5098.
   מטפל בבקשות HTTP נכנסות בצורה א-סינכרונית; כל בקשה מטופלת ב-Thread
   נפרד מתוך ThreadPool המנוהל על ידי ה-CLR.

.2 תהליך הרצת האלגוריתם — בעת הרצת משימת שיבוץ (POST /api/packingjob/{id}/run),
   האלגוריתם BranchAndBoundSolver רץ בתהליך ה-API עצמו; מגבלת הזמן של
   300 שניות מנוהלת על ידי Stopwatch פנימי.

.3 תהליך SQL Server LocalDB (SqlLocalDB.exe) — תהליך נפרד של מנוע SQL Server
   המופעל אוטומטית עם חיבור ראשון דרך ADO.NET, ומאזין על Named Pipe מקומי.

.4 תהליך הדפדפן (Browser) — תהליך הלקוח. רץ על מכונת הלקוח, שולח
   בקשות HTTP ומרנדר את תוצאות Three.js ב-WebGL Thread ייעודי.

.5 תהליכי בדיקות (dotnet test) — תהליך xUnit המריץ 20 בדיקות יחידה
   (BoxTests, BranchAndBoundTests, GreedySolverTests, LowerBoundTests)
   ללא צורך בחיבור לשרת או למסד הנתונים.


11.4 ארכיטקטורת רשת

המערכת פועלת כ-Single-Server Architecture — שרת ה-API מגיש גם את דפי
הלקוח (UseStaticFiles) וגם עונה לקריאות REST, על פורט 5098.

  [דפדפן לקוח]
       ↕  HTTP/REST  (TCP פורט 5098)
  [שרת OptiLoad.API — localhost:5098]
       ↕  Named Pipe / TCP מקומי
  [SQL Server LocalDB — OptiLoadDB]

מסד הנתונים LocalDB רץ בתהליך מקומי על אותה המכונה ואינו מצריך
תעבורת רשת חיצונית. בסביבת פרודקשן ניתן להחליפו ב-SQL Server מלא
תוך שינוי ConnectionString בלבד, ללא שינוי בקוד.


11.5 תיאור ה-API בארכיטקטורה

ה-API מחולק ל-8 בקרים; כל בקר אחראי לתחום ייעודי:

  בקר                   | Endpoint לדוגמה                    | שיטה   | הגנה
  -----------------------|------------------------------------|--------|--------
  AuthController         | /api/auth/login                    | POST   | ציבורי
  BoxController          | /api/box  /  /api/box/{id}         | GET/POST/PUT/DELETE | JWT
  ContainerController    | /api/container                     | GET/POST/PUT/DELETE | JWT
  PackingJobController   | /api/packingjob/{id}/run           | POST   | JWT
  VisualizationController| /api/visualization/{jobId}         | GET    | JWT
  SnapshotController     | /api/snapshot/{jobId}              | GET/POST | JWT
  SessionController      | /api/session  /  /api/session/{token} | GET/POST | מעורב
  ExportController       | /api/export/{jobId}                | GET    | JWT

כל ה-Endpoints המוגנים מחייבים כותרת:
  Authorization: Bearer <JWT_TOKEN>

נקודת הכניסה הציבורית היחידה היא /api/auth/login. בנוסף,
/api/session/{token} ציבורי לגישת משתמשי שטח ע"י LinkToken.
LoginRateLimiter מגביל ניסיונות כניסה ל-3 ניסיונות / 5 דקות לפי IP.


11.6 תיאור פרוטוקולי התקשורת

● HTTP/1.1 + REST — פרוטוקול התקשורת בין הלקוח לשרת. כל משאב
  מיוצג ב-URL ייחודי עם פעלי GET / POST / PUT / DELETE; תגובות
  בפורמט JSON; קודי מצב HTTP סטנדרטיים (200, 201, 400, 401, 404).

● JWT Bearer Token — פרוטוקול האימות. לאחר כניסה מוצלחת מונפק
  טוקן חתום ב-HMAC-SHA256 עם תפוגה של שעתיים; מועבר בכותרת
  Authorization בכל בקשה מוגנת; כולל Claim של UserId למניעת IDOR.

● ADO.NET / TDS — פרוטוקול התקשורת בין שכבת ה-Data למסד SQL Server
  LocalDB, מבוצע דרך Microsoft.Data.SqlClient. כל פרמטר מועבר
  כ-SqlParameter ולא כחלק מהמחרוזת, למניעת SQL Injection.


11.7 שרת-לקוח — תקשורת ומחזור בקשה-תגובה

התקשורת בין הלקוח לשרת מבוססת על RESTful API; להלן מחזור בקשה
מלא של הרצת משימת שיבוץ:

.1 הלקוח שולח POST /api/packingjob/{id}/run דרך Fetch API עם JWT Token
   בכותרת Authorization.
.2 Middleware בשרת מאמת את הטוקן; LoginRateLimiter בודק את ה-IP.
.3 PackingJobController קורא ל-PackingService שמשיג את נתוני הארגזים
   והקונטיינר ממסד הנתונים דרך IPackingRepository.
.4 PackingService מריצה HeuristicH1 → BranchAndBoundSolver → GreedySolver
   (במידת הצורך) ומחזירה PackingResult.
.5 התוצאה נשמרת כ-PlacementResults במסד הנתונים.
.6 השרת מחזיר תגובת JSON עם מיקומי הארגזים, מספר קונטיינרים
   שנוצלו וסמן IsOptimal ללקוח.
.7 הלקוח מקבל את הנתונים ומרנדר את התצוגה התלת-ממדית ב-Three.js.

12. ניתוח ותרשים UML / Use Cases של המערכת המוצעת

12.1 תיאור ה-UC העיקריים של המערכת

המערכת OptiLoad מגדירה שני סוגי שחקנים: מנהל מערכת (Admin) — בעל הרשאות מלאות
לניהול, שיבוץ וייצוא; ומשתמש שטח (Guest) — בעל גישת צפייה בלבד דרך קוד סשן
שהופץ לו. המערכת תומכת בעשרה Use Cases עיקריים המחולקים לשלוש קבוצות:

קבוצה א' — ניהול נתונים:
.1 UC-01: כניסה למערכת — אימות מנהל באמצעות שם משתמש וסיסמה קבלת JWT Token.
.2 UC-02: ניהול ארגזים — הוספה, עריכה ומחיקה של ארגזים עם כל פרמטריהם.
.3 UC-03: הגדרת קונטיינר — יצירת תבנית קונטיינר עם ממדים וכושר נשיאה.

קבוצה ב' — שיבוץ ותצוגה:
.4 UC-04: יצירת משימת שיבוץ — שיוך ארגזים לקונטיינר ויצירת PackingJob.
.5 UC-05: הרצת אלגוריתם — הפעלת מנוע B&B / Greedy וקבלת תוצאות מיקום.
.6 UC-06: צפייה בתוצאות תלת-ממד — הצגה אינטראקטיבית של תוצאות השיבוץ.
.7 UC-07: ניהול תמונות מצב — שמירת Snapshots לכל שלב טעינה וצפייה בהם.

קבוצה ג' — שיתוף וייצוא:
.8 UC-08: ניהול סשנים — יצירת סשן משותף וקבלת קוד הצטר.
.9 UC-09: צפייה בסשן — כניסת משתמש שטח לצפייה ללא חשבון מנהל.
.10 UC-10: ייצוא תכנית טעינה — הורדת תכנית הטעינה כקובץ JSON.

תנאי יציאה: משתמש השטח רואה את תכנית הטעינה.

──────────────────────────────────────────────────────
UC-10 | ייצוא תכנית טעינה
──────────────────────────────────────────────────────
שחקן:     מנהל מערכת
תנאי קדם: PackingJob הושלם; בעלות מאומתת
זרימה:
  .1 המנהל לוחץ "ייצוא JSON" — GET /api/export/{jobId}.
  .2 השרת מאמת שה-JobId שייך למנהל המחובר (בדיקת IDOR).
  .3 נוצר קובץ JSON הכולל: מידות קונטיינר, רשימת שלבי
     טעינה (כל שלב: שם ארגז, מיקום X/Y/Z, ממדים, כיוון),
     אחוז ניצול נפח, ותמונות מצב.
  .4 הקובץ מוחזר ל-Browser להורדה.
תנאי יציאה: קובץ JSON הורד בהצלחה עם תכנית הטעינה המלאה.


12.3 מבנה נתונים

המערכת עושה שימוש במבני נתונים הבאים, ולכל אחד נימוק לבחירתו:

  מבנה נתונים        | מיקום בקוד              | שימוש ונימוק
  -------------------|-------------------------|---------------------------------------------
  List<PlacedBox>    | PackingState            | רשימה דינמית של ארגזים שמוקמו בקונטיינר.
                     |                         | נבחרה כי B&B מוסיף ומסיר פריטים בגב המחסנית
                     |                         | (RemoveLastBox = _placed.RemoveAt(Count-1)),
                     |                         | מה שמחקה מחסנית (LIFO) על גבי רשימה.
  -------------------|-------------------------|---------------------------------------------
  List<BoxInstance>  | BranchAndBoundSolver,   | רשימה דינמית של כל מופעי הארגז בהתחשב
                     | HeuristicH1, Greedy     | בכמויות. נבחרה על פני מערך קבוע כי
                     |                         | הכמות אינה ידועה מראש.
  -------------------|-------------------------|---------------------------------------------
  HashSet<string>    | BranchAndBoundSolver    | מזהה אם ארגז מסוים כבר מוקם בחיפוש.
                     |                         | זמן גישה O(1) לעומת O(n) ב-List<>,
                     |                         | קריטי לביצועים בחיפוש עם 10M+ צמתים.
  -------------------|-------------------------|---------------------------------------------
  HashSet<(W,H,D)>   | Box.GetAllowedRotations | מונע כפילות בין כיוונים זהים (למשל ארגז
                     |                         | מרובע שכל סיבוביו זהים). O(1) לבדיקת שייכות.
  -------------------|-------------------------|---------------------------------------------
  IEnumerable<T>     | ממשקי Repository,       | הזרמת נתונים עצלה (Lazy Evaluation) ללא
                     | אלגוריתמים              | טעינת כלל הרשימה לזיכרון בו-זמנית.
  -------------------|-------------------------|---------------------------------------------
  עץ רקורסיבי        | BranchAndBoundSolver    | עץ החיפוש של B&B מיוצג כ-DFS רקורסיבי
  (Implicit Stack)   |                         | תוך שימוש במחסנית הקריאות של ה-CLR.
                     |                         | הדבר חוסך מבנה תור/מחסנית מפורש ומפשט
                     |                         | את כתיבת הניסיון-וחזרה (Backtracking).


12.4 הקשרים בין היחידות השונות

זרימת התלויות בין יחידות המערכת נקבעת על פי עקרון ה-Dependency Inversion:

  יחידה              | תלויה ב-               | מיושמת על ידי
  -------------------|------------------------|----------------------------------------
  OptiLoad.API       | OptiLoad.Core (ממשקים) | קורא לממשקי IPackingRepository וכד'
                     | OptiLoad.Data          | Dependency Injection בזמן ריצה
  OptiLoad.Data      | OptiLoad.Core (ממשקים) | DatabaseService מממש IPackingRepository,
                     |                        | IAdminRepository, ISnapshotRepository;
                     |                        | DatabaseSessionService מממש ISessionRepository
  OptiLoad.Core      | ——                     | אין תלויות חיצוניות; ריבוי ממשקי
                     |                        | Repository מאפשר החלפת שכבת Data
                     |                        | ללא שינוי ב-Core
  OptiLoad.Tests     | OptiLoad.Core          | בודק ישירות את אלגוריתמי Core
                     |                        | ללא DB וללא HTTP

קשרי מחלקות פנימיות ב-Core:

● BranchAndBoundSolver מכיל מופע של HeuristicH1, LowerBoundCalculator
  ו-GreedySolver — קריאה ישירה (Composition).
● HeuristicH1 משתמש ב-SingleBinFiller לניסיונות המיון.
● CornerPointsFinder מקבל PackingState ומחזיר רשימת Position3D — ממשק נקי
  ללא תלות בשאר האלגוריתמים.
● GravitySettler מקבל PackingState ומשנה את מיקום הארגזים אחרי כל שלב.
● LoadingSequencer מקבל List<PlacedBox> ומחזיר רשימה ממוינת לפי
  מיון טופולוגי (Topological Sort).
● PackingService מקשרת בין ה-API לאלגוריתמים דרך IPackingRepository —
  לא קוראת ל-BranchAndBoundSolver ישירות, אלא בונה מופע ומזריקה
  את הנתונים.

12.11 תיאור המחלקות המוצעות

להלן תיאור מפורט של המחלקות העיקריות בפרויקט, מחולק לפי שכבות:

─────────────────────────────────────────
שכבת Domain — מודלי הדומיין (OptiLoad.Core)
─────────────────────────────────────────

Box
  תפקיד: מייצגת הגדרת ארגז פיזי עם כל תכונותיו.
  קלט:   נתונים שמנהל הזין בממשק (שם, מידות, משקל, שביר, סיבוב).
  פלט:   מופעי Rotation לכל כיוון מותר, ערך Volume מחושב.
  שדות עיקריים: BoxId, BoxName, Width, Height, Depth, WeightKg,
    IsFragile, AllowRotation.
  פונקציה מרכזית: GetAllowedRotations() — מחזירה עד 6 כיוונים
    ייחודיים; ארגז שסיבובו אסור מחזיר כיוון אחד בלבד.

ContainerTemplate + Container
  תפקיד: ContainerTemplate מגדיר תבנית עם ממדים וכושר נשיאה;
    Container הוא מופע בודד שנוצר על בסיס תבנית עם קוד וסטטוס.
  קלט:   נתוני תבנית (Width, Height, Depth, MaxWeightKg).
  פלט:   Container עם ContainerStatus (Available / Loading / Closed).

PackingJob
  תפקיד: מייצגת משימת שיבוץ מרגע יצירה עד השלמה.
  קלט:   AdminId, ContainerId, רשימת ארגזים עם כמויות.
  פלט:   BinsUsed, VolumeUtilization, TotalWeightKg, SolveTimeSeconds,
    IsOptimal, StatusMessage.
  Enum JobStatus: Pending → Running → Completed / Failed.

PlacementResult
  תפקיד: שומרת את תוצאת המיקום של ארגז בודד בתוך קונטיינר.
  קלט:   JobId, BoxId, InstanceIndex, BinIndex.
  פלט:   PosX/PosY/PosZ (נקודת הפינה השמאלית-תחתונה-קדמית),
    PlacedWidth/Height/Depth, RotationIndex.

PackingSession + SessionUser + SessionBox
  תפקיד: PackingSession מייצגת סשן צפייה משותף עם LinkToken ייחודי.
    SessionUser שומרת מי הצטרף לסשן; SessionBox קושרת את
    הארגזים שהוצגו בסשן.
  קלט:   AdminId, Name, Description.
  פלט:   LinkToken (GUID), Status (Open/Closed).

─────────────────────────────────────────
מחלקות חישוב (ComputationModels)
─────────────────────────────────────────

ContainerDimensions
  תפקיד: ייצוג נקי של ממדי הקונטיינר כאובייקט חישובי (נפרד
    מ-Container שהוא מודל DB). מכיל גם Volume מחושב.
  קלט:   Width, Height, Depth, MaxWeightKg.
  פלט:   Volume (W×H×D).

BoxInstance
  תפקיד: מופע בודד של ארגז בתוך משימה. ה-BoxDefinition הוא
    הגדרת הארגז, ו-InstanceIndex הוא מספר הסידורי של המופע
    (לדוגמה: ארגז "A" עם כמות 3 יצור BoxInstance #0, #1, #2).
  פלט:   InstanceId (שם#מספר) לזיהוי ייחודי בחיפוש.

Rotation (record struct)
  תפקיד: מייצגת כיוון ספציפי של ארגז (W×H×D אחרי הסיבוב +
    מספר אינדקס). record struct — ערך-בלתי-ניתן-לשינוי,
    השוואה לפי ערכים.

Position3D (record struct)
  תפקיד: קואורדינטות (X, Y, Z) של נקודת מיקום. record struct
    לאותה סיבה — ערכים מספריים שמשווים אחד-לאחד.

PlacedBox
  תפקיד: ארגז שכבר מוקם — כולל BoxInstance, Position3D ו-Rotation.
    מחשב X1/Y1/Z1 (פינה קרובה) ו-X2/Y2/Z2 (פינה רחוקה).
  פונקציה מרכזית: OverlapsWith(other) — בדיקת חפיפה תלת-ממדית
    בזמן O(1) על פי AABB (Axis-Aligned Bounding Box).

PackingState
  תפקיד: מייצגת מצב מיידי של קונטיינר במהלך החיפוש — רשימת
    PlacedBoxes שמוקמו עד כה, משקל כולל וסה"כ נפח.
  פונקציות: AddBox / RemoveLastBox (לתמיכה ב-Backtracking),
    Clone() (לשמירת המצב הטוב ביותר).

PackingResult
  תפקיד: תוצאה סופית מהאלגוריתם — PlacedBoxes, UnplacedBoxes,
    BinsUsed, VolumeUtilization, IsOptimal, SolveTimeSeconds.

─────────────────────────────────────────
שכבת Algorithms (OptiLoad.Core)
─────────────────────────────────────────

BranchAndBoundSolver
  תפקיד: אלגוריתם החיפוש המרכזי — מחפש את מספר הקונטיינרים
    המינימלי לאריזת כלל הארגזים.
  קלט:   ContainerDimensions, IEnumerable<BoxInstance>.
  פלט:   PackingResult (מיקומים, BinsUsed, IsOptimal).
  מנגנון: DFS רקורסיבי עם Pruning לפי L0/L1/L2, מגבלת
    10M צמתים ו-300 שניות; Greedy Fallback אוטומטי.

HeuristicH1
  תפקיד: מייצרת פתרון פתיחה מהיר לפני ה-B&B.
  קלט:   ContainerDimensions, List<BoxInstance>.
  פלט:   (binsUsed, placements) — הפתרון הטוב מבין 3 מיונים.
  מנגנון: מריצה SingleBinFiller בשלושה סדרי מיון שונים
    (לפי Depth / Width / Height) ובוחרת את ניצול הנפח הגבוה ביותר.

GreedySolver
  תפקיד: אלגוריתם חמדני — ממקם ארגזים לפי סדר ובחירת
    נקודת הפינה הראשונה הזמינה.
  קלט:   ContainerDimensions, IEnumerable<BoxInstance>.
  פלט:   PackingResult; מורכבות O(n × corners × rotations).
  שימוש: Fallback כשה-B&B חורג ממגבלת זמן/צמתים.

SingleBinFiller
  תפקיד: ממלא קונטיינר בודד עם B&B מקומי; משמש ב-HeuristicH1.
  קלט:   ContainerDimensions, List<BoxInstance> לניסיון.
  פלט:   רשימת מיקומים בתוך קונטיינר בודד.

CornerPointsFinder
  תפקיד: מחשב את נקודות המיקום החוקיות ("נקודות פינה")
    בחלל הקונטיינר.
  קלט:   PackingState (ארגזים שכבר מוקמו), ContainerDimensions.
  פלט:   List<Position3D> — כל נקודה שבה ניתן למקם ארגז.
  עיקרון: נקודת פינה נוצרת בכל צומת של ארגז קיים — X2, Y1, Z1
    בכל קומבינציה — כדי למקסם ניצול חלל.

GravitySettler
  תפקיד: מדמה כבידה — מוריד ארגזים שמרחפים לגובה הנמוך ביותר
    האפשרי אחרי כל שלב מיקום.
  קלט:   PackingState, ContainerDimensions.
  פלט:   PackingState מעודכן עם מיקומי Y מוקטנים.

LowerBoundCalculator
  תפקיד: מחשב חסמים תחתונים לגזם ענפים בחיפוש.
  קלט:   List<BoxInstance>, ContainerDimensions.
  פלט:   int — מספר הקונטיינרים המינימלי האפשרי.
  שלוש שיטות: L0 (נפח טהור), L1 (פיצול ציר), L2 (חסם מחוזק).

LoadingSequencer
  תפקיד: קובע את סדר הטעינה הפיזי לצוות השטח.
  קלט:   List<PlacedBox>, כיוון הדלת (LoadingFace).
  פלט:   List<PlacedBox> ממוינת לפי מיון טופולוגי — הארגז
    הרחוק ביותר מהדלת נטען ראשון.

─────────────────────────────────────────
שכבת Services (OptiLoad.Core)
─────────────────────────────────────────

PackingService
  תפקיד: אורקסטרטור — מתאמת בין ה-API לאלגוריתם.

6. רקע תיאורטי

פרק זה סוקר את הרקע התיאורטי הנחוץ להבנת הבעיה האלגוריתמית שבבסיס פרויקט
OptiLoad. הסקירה מתחילה בהגדרת הדומיין הפיזי — המכולה ועבודת הטעינה — ממשיכה
ביסודות תורת הסיבוכיות, ומפרטת את הבעיות הקלאסיות שהפרויקט נותן להן מענה, לצד
הגישות האלגוריתמיות שנבחרו.


6.1 מכולה ועבודת טעינה — הדומיין הפיזי

מהי מכולה?

המכולה (Shipping Container) היא תיבת מתכת תקנית המשמשת לשינוע סחורות בין אמצעי
תחבורה שונים — אוניות, משאיות ורכבות — ללא צורך בפריקה ועמסה חוזרת של המטען בכל
מעבר. הרעיון פותח לראשונה בשנות החמישים של המאה העשרים על ידי מאלקולם מקלין, ומאז
הפך לתשתית הבסיסית של הסחר הימי הבינלאומי; היום כ-90% מהסחר העולמי מועבר במכולות
(Levinson, The Box: How the Shipping Container Made the World Smaller, 2006).

תקן ISO 668 מגדיר את ממדי המכולה הסטנדרטית הנפוצה ביותר:

  סוג          | אורך           | רוחב    | גובה    | כושר נשיאה
  -------------|----------------|---------|---------|----------------
  20-foot TEU  | 6.06 מטר       | 2.44 מ' | 2.59 מ' | ~28,000 ק"ג
  40-foot FEU  | 12.19 מטר      | 2.44 מ' | 2.59 מ' | ~26,000 ק"ג
  High Cube    | 12.19 מטר      | 2.44 מ' | 2.89 מ' | ~26,000 ק"ג

הנפח הפנימי של מכולת 20 רגל עומד על כ-33 מטר מעוקב. בשל אחידות ממדיה, המכולה
מאפשרת תכנון מוקדם של סידור המטען — וזוהי בדיוק הבעיה שהפרויקט בא לפתור.

מהי עבודת טעינה?

עבודת טעינה (Container Loading) היא תהליך שיבוץ ארגזים ויחידות מטען בתוך מכולת
השינוע. למרות שהתהליך נראה פשוט על פניו, הוא כרוך בשיקולים מורכבים הנדרשים
בו-זמנית:

● שיקולי מקום — סידור ארגזים בגדלים שונים כך שמרחב החלל המנוצל יהיה מקסימלי
  ולא ייוותרו "חורים" ריקים בלתי שמישים.
● שיקולי משקל — חלוקת העומס חייבת לעמוד בכושר הנשיאה הכולל, ויש לשמור על
  יציבות מרכז הכובד לאורך ציר המכולה.
● שיקולי שבירות — ארגזים מסומנים כשבירים (Fragile) אינם יכולים לשאת עומס
  מעליהם; לכן הם חייבים להיות ממוקמים בשכבה העליונה של הסידור.
● שיקולי נגישות — ארגזים המיועדים לפריקה ראשונה בנקודת היעד צריכים להיות
  ממוקמים בסמוך לדלת המכולה; ארגזים לתחנות מאוחרות נדחפים פנימה.

כיום מרבית עבודות הטעינה מתבצעות ידנית על בסיס ניסיון העובד וראייתו המרחבית.
היעדר תכנון אלגוריתמי מסודר מוביל בדרך כלל לניצול חסר של נפח המכולה ולשגיאות
בטיפול בסחורה רגישה. מחקרים בתחום מדווחים על בזבוז ממוצע של 15%-25% מנפח
המכולה בטעינות ידניות לא מתוכננות (Bischoff & Ratcliff, Issues in the Development
of Approaches to Container Loading, 1995).


6.2 מורכבות חישובית — P, NP ו-NP-Hard

הבנת האתגר החישובי שבבסיס בעיית האריזה דורשת היכרות עם יסודות תורת הסיבוכיות.

בתורת הסיבוכיות, בעיות חישוביות מסווגות לפי מידת הקושי לפתור אותן. מחלקת P
(Polynomial Time) מגדירה בעיות שניתן לפתור בזמן פולינומי ביחס לגודל הקלט —
כלומר ככל שגדל הקלט, זמן הריצה גדל בצורה מתונה וצפויה (O(n^k) עבור קבוע k
כלשהו). מחלקת NP (Nondeterministic Polynomial) מגדירה בעיות שניתן לאמת את
נכונות פתרון נתון בזמן פולינומי — אך מציאתו מלכתחילה עשויה לדרוש זמן
אקספוננציאלי (Garey & Johnson, Computers and Intractability: A Guide to the
Theory of NP-Completeness, 1979).

NP-Hard הן הקשות ביותר בתוך NP — בעיות שכל בעיה ב-NP ניתנת לצמצום (Reduction)
אליהן בזמן פולינומי. המשמעות המעשית היא חמורה: ככל שגדל מספר הפריטים בקלט,
מספר הצירופים האפשריים גדל בצורה אקספוננציאלית. לדוגמה, עבור 20 ארגזים בלבד,
מספר סידורי המיקום האפשריים עולה על מיליארד — ובדיקתם כולם אינה ישימה בזמן
סביר.

לכן בעיות NP-Hard נפתרות בפרקטיקה באחת משתי גישות:
.1 אלגוריתמים מדויקים עם קיצוץ (כגון Branch and Bound) — מבטיחים אופטימליות
   אך עם מגבלת זמן ריצה.
.2 אלגוריתמים היוריסטיים ומטא-היוריסטיים — מספקים פתרון טוב-מספיק בזמן ריצה
   קביל, ללא ערובה לאופטימליות מוחלטת.


6.3 בעיית האריזה התלת-ממדית (3D Bin Packing Problem)

בעיית האריזה התלת-ממדית (3D-BPP — Three-Dimensional Bin Packing Problem) היא
אחת הבעיות הקלאסיות בתחום האופטימיזציה הקומבינטורית. הבעיה עוסקת בשאלה: נתונה
קבוצה של פריטים תלת-ממדיים בעלי ממדים שונים — כיצד לארוז אותם בתוך מיכל (Bin)
בעל נפח מוגבל, כך שהנפח הזמין ינוצל בצורה מיטבית? (Martello, Pisinger & Vigo,
Three-Dimensional Bin Packing Problems, 2000).

בגרסה הכללית של הבעיה מוגדרים:

● כל ארגז i על ידי שלשה ממדים: (wᵢ, hᵢ, dᵢ) — רוחב, גובה ועומק.
● לכל ארגז משקל mᵢ ומאפיין שבירות (IsFragile).
● המכולה מוגדרת על ידי (W, H, D) וכושר נשיאה מקסימלי M.

מטרת האופטימיזציה: למצוא עבור כל ארגז מיקום (x, y, z) בתוך המכולה כך שיתקיימו
כל האילוצים הבאים בו-זמנית:

.1 אי-חפיפה — לא קיימים שני ארגזים החולקים נפח משותף בחלל התלת-ממדי.
.2 גבולות — כל ארגז נמצא כולו בתוך גבולות המכולה:
   0 ≤ x + wᵢ ≤ W,  0 ≤ y + hᵢ ≤ H,  0 ≤ z + dᵢ ≤ D.
.3 מגבלת משקל — Σ mᵢ ≤ M.
.4 אילוץ שבירות — ארגז שביר אינו יכול לשאת עליו שום עומס (y + hᵢ = שיא ה-Y
   של הסידור מעליו ריק).

הגרסה הנפוצה ביותר בספרות נקראת 3D-BPP "אורתוגונלית" — כלומר כל ארגז ממוקם
בציר ישר ואינו מוטה. גרסה מורחבת מתירה סיבובים (Rotations) — עד שש אוריינטציות
שונות לכל ארגז. שתי הגרסאות הוכחו כ-NP-Hard (Garey & Johnson, 1979), אך
הגרסה עם סיבובים קשה עוד יותר בפרקטיקה בשל ריבוי מרחב החיפוש.

בפרויקט OptiLoad מומשה הגרסה עם סיבובים (עד 6 כיוונים ייחודיים לכל ארגז,
בהתאם לתכונת AllowRotation).


6.4 בעיית תיק הנשיאה (Knapsack Problem) וקשרה לפרויקט

בעיית תיק הנשיאה (Knapsack Problem) היא בעיית אופטימיזציה קלאסית ורלוונטית
ישירות לפרויקט. בצורתה הבסיסית: נתונה קבוצה של n פריטים שלכל אחד מהם משקל
wᵢ וערך vᵢ, ותיק בעל קיבולת W. המטרה היא לבחור תת-קבוצה של פריטים כך
שסך הערך יהיה מקסימלי מבלי לחרוג ממגבלת המשקל (Kellerer, Pferschy & Pisinger,
Knapsack Problems, 2004):

  Maximize  Σ vᵢxᵢ     subject to:  Σ wᵢxᵢ ≤ W,  xᵢ ∈ {0,1}

הקשר לבעיית האריזה התלת-ממדית ישיר: ה"ערך" של כל ארגז הוא נפחו התורם לניצול
המכולה (vᵢ = wᵢ × hᵢ × dᵢ), וה"אילוץ" הוא שלושת ממדי המכולה בו-זמנית. גרסת
תלת-ממד של הבעיה — המוכרת כ-3D Knapsack — מהווה בסיס תיאורטי לגישת החסמים
התחתונים (Lower Bounds) שיושמה ב-OptiLoad:

● חסם L0 — חסם נפח טהור: מספר הקונטיינרים המינימלי הוא לפחות ⌈ΣVᵢ / V_container⌉.
● חסם L1 — חסם Dual Feasibility: לוקח בחשבון גם את ממד הארגזים לאורך ציר
  אחד, ומחשב חסם מחמיר יותר.
● חסם L2 — חסם מחוזק (Strengthened Bound): מחלק את הארגזים לקבוצות על פי
  ממדיהם ביחס לממד המכולה, ומספק את החסם הקפדני ביותר מבין השלושה.

גם בעיית תיק הנשיאה הוכחה כ-NP-Hard בצורתה הכללית.


6.5 אלגוריתמים לפתרון — גישות שיושמו בפרויקט

6.5.1 אלגוריתם חמדן (Greedy Algorithm)

אלגוריתם חמדן הוא גישה אלגוריתמית שבה הפתרון נבנה שלב אחרי שלב, כאשר בכל שלב
נבחרת האפשרות הטובה ביותר מנקודת מבט מקומית — מבלי להתחשב בהשפעת הבחירה על
שלבים עתידיים (Cormen, Leiserson, Rivest & Stein, Introduction to Algorithms,
4th ed., 2022). לאלגוריתמים מסוג זה מורכבות זמן נמוכה, בדרך כלל O(n log n)
עקב שלב המיון.

כיצד עובד האלגוריתם בפרויקט OptiLoad:
.1 מיון — הארגזים ממוינים בסדר יורד לפי נפחם, כך שהגדולים מטופלים ראשונים.
.2 בחירת מיקום — עבור כל ארגז מחפשים את נקודת הפינה (Corner Point) הראשונה
   הזמינה בחלל הקונטיינר שמתאימה לו מבחינת ממדים ומשקל.
.3 בדיקת אילוצים — ארגז שביר יוצב בחלק העליון של המכולה בלבד; לא ייצב עליו
   ארגז אחר; נבדקת חפיפה עם כל ארגז שמוקם כבר.
.4 עדכון המצב — לאחר שיבוץ כל ארגז מתעדכן PackingState ונחושבות נקודות הפינה
   החדשות.

יתרון מרכזי: מהירות ריצה גבוהה — O(n × corners × rotations) — המאפשרת תמיד
קבלת פתרון בזמן קביל גם לקלטים גדולים. חיסרון מרכזי: אינו מבטיח אופטימליות
מוחלטת — בחירה מקומית "חמדנית" עלולה לחסום מיקומים טובים לארגזים מאוחרים.

בפרויקט זה האלגוריתם החמדן ממלא שני תפקידים: HeuristicH1 משתמשת בו לבניית
פתרון פתיחה מהיר (Initial Solution) המצמצם את מרחב החיפוש של B&B; ו-GreedySolver
משמש כ-Fallback אוטומטי כאשר B&B מגיע למגבלת הזמן או הצמתים.


6.5.2 אלגוריתם Branch and Bound

Branch and Bound (B&B) הוא אלגוריתם חיפוש מדויק המשמש לפתרון בעיות אופטימיזציה
קומבינטורית. האלגוריתם בונה עץ חיפוש שבו כל צומת מייצג מצב חלקי של הפתרון,
ומכיל מנגנון "קיצוץ" (Pruning) המאפשר לוותר על ענפים שלא יכולים להוביל לפתרון
טוב יותר מהידוע (Lawler & Wood, Branch and Bound Methods: A Survey, 1966).

שלבי הפעולה:
.1 פיצול (Branch) — הצומת הנוכחי מפוצל: עבור ארגז שטרם מוקם, נוצרים צמתים
   ילדים לכל מיקום וכיוון אפשרי.
.2 חסימה (Bound) — לכל צומת מחושב חסם תחתון (Lower Bound) — אומדן מינימלי
   לכמות הקונטיינרים הנדרשת מנקודה זו ואילך. בפרויקט נבדקים L0, L1 ו-L2.
.3 קיצוץ (Pruning) — אם החסם התחתון של ענף גדול או שווה למספר הקונטיינרים
   בפתרון הטוב הידוע, הענף כולו נפסל ואינו נחקר — חיסכון מהותי בזמן.
.4 עדכון הפתרון הטוב — בכל פעם שמגיעים לפתרון שלם ותקין הדורש פחות קונטיינרים
   מהפתרון הידוע, הוא מתעדכן ומשמש כסף חדש לקיצוץ.

ב-OptiLoad ה-B&B מוגבל ל-10,000,000 צמתים ו-300 שניות כדי להבטיח שהמערכת
תמיד תחזיר תגובה בזמן סביר. כאשר מגיעים למגבלה — הפתרון הטוב שנמצא עד כה
מסומן כ-IsOptimal = false, ו-GreedySolver משלים את הארגזים שלא מוקמו.

יתרון מרכזי: כאשר האלגוריתם מסתיים לפני המגבלה, הפתרון המתקבל מובטח אופטימלי
— מספר הקונטיינרים המינימלי האפשרי. חיסרון: במקרה הגרוע ביותר, כאשר הקיצוץ
מועט, זמן הריצה הוא אקספוננציאלי; לכן חיוני שה-HeuristicH1 תספק נקודת פתיחה
טובה המצמצמת משמעותית את עץ החיפוש.


6.6 סיכום הרקע התיאורטי

בעיית שיבוץ הארגזים במכולה — בצורתה השלמה הכוללת ממדים, משקל, שבירות וסיבובים
— היא בעיה NP-Hard מורכבת שאינה ניתנת לפתרון אופטימלי מובטח בזמן פולינומי.
המחקר בתחום מצביע על שילוב של שלוש שיטות כאסטרטגיה יעילה: חסמים תחתונים חזקים
לקיצוץ מהיר, פתרון פתיחה היוריסטי איכותי, ואלגוריתם B&B מוגבל-זמן כגישת החיפוש
המדויק.

גישה היברידית זו — HeuristicH1 ← BranchAndBound ← GreedyFallback — מאפשרת
למערכת OptiLoad לחזור תמיד בתוצאה שמישה בזמן ריצה קביל, תוך מקסום הסיכוי
לפתרון האופטימלי כאשר הבעיה ניתנת לפתרון בתוך מגבלת הזמן.

המשתמש יכול לצפות בארגזים המשויכים לתוכנית ולהוסיף בקשות להוספת ארגזים
— ללא הרשאות כתיבה לנתוני הליבה ומבלי לדרוש JWT Token.

17.6.2 תיאור המסך

  סרגל עליון:
    — לוגו "🚛 OptiLoad".
    — תגית שם התוכנית (Session Badge) — מוצגת לאחר אימות הטוקן.

  שלב 1 — טופס הצטרפות (לפני אימות):
    — שדה הזנת LinkToken.
    — שדה "שם" (זיהוי האיש בשטח).
    — כפתור "הצטרף לתוכנית".
    — הודעת שגיאה (LinkToken שגוי / תוכנית סגורה).
    — ספינר טעינה.

  שלב 2 — תצוגת תוכנית (לאחר אימות):
    — כותרת "ניהול ארגזים — [שם התוכנית]".
    — טבלת ארגזים: שם, W, H, D, משקל, כמות — נתוני קריאה בלבד.
    — סקציית "הוספת ארגז לבקשה":
        • רשימה נפתחת (Dropdown) של סוגי הארגזים הקיימים בתוכנית.
        • שדה כמות.
        • כפתור "הוסף" (ירוק).
    — הגדרות: הזיהוי נשמר ב-sessionStorage לפי מפתח ייחודי לכל LinkToken.

    משך הנעילה: 5 דקות.

  "לא ניתן להתחבר לשרת"
    מתי: שגיאת רשת (fetch נכשל, השרת לא זמין).
    מראה: תיבה אדומה.

20. ממשק המשתמש

ממשק מערכת OptiLoad תוכנן סביב שני עקרונות מרכזיים: פשטות תפעולית
ומינימום עומס קוגניטיבי על המשתמש בשטח.

הממשק כהה ומינימליסטי — עיצוב המבוסס על כרטיסים ברורים עם הדגשות
צבעוניות בלבד למצבים חשובים (שגיאה, הצלחה, פעולה). אין עומס ויזואלי
מיותר; כל אלמנט על המסך מייצג פעולה ספציפית.

המערכת מבחינה בין שני סוגי משתמשים ומציגה לכל אחד בדיוק את מה שרלוונטי
לו: מנהל המערכת רואה ממשק עשיר עם שליטה מלאה; משתמש השטח רואה ממשק
מינימלי — שדה טוקן, רשימת ארגזים, ושורת הוספת בקשה. לא יותר.

הניווט בין מסכי המנהל מתבצע דרך סרגל עליון קבוע, כך שמיקום המשתמש
תמיד ברור ועוברים בין מסכים בלחיצה אחת. שגיאות אימות (JWT פג תוקף)
מטפלות בעצמן — ניתוב אוטומטי לדף הכניסה ללא הודעת שגיאה טכנית.

כל הממשק כתוב בעברית מלאה עם תמיכת RTL, ללא תלות בספריות UI חיצוניות.

21. תיאור תהליכי אבטחת המידע

פרק זה מתאר את מנגנוני אבטחת המידע המיושמים במערכת OptiLoad,
בהתאם לעקרון Defense in Depth.


21.1 תיאור ההתקפות וההגנות

  וקטור תקיפה           | הגנה
  ----------------------|----------------------------------------------------------
  Brute Force           | LoginRateLimiter: נעילה ל-5 דקות לאחר 3 כשלונות;
                        | 429 + Retry-After; ספירה לאחור בממשק
  SQL Injection         | SqlCommand עם SqlParameter בלבד — אין שרשור מחרוזות
  IDOR                  | בדיקת AdminId בכל גישה לסשן/Snapshot/Job — Forbid אם אינו הבעלים
  גישה ללא אימות       | [Authorize] על כל ה-Controllers; ניתוב אוטומטי ל-login ב-401
  עקיפת נעילת IP      | נורמליזציית IPv6→IPv4 (MapToIPv4) — כתובת אחת ב-Dictionary, לא שתיים
  Information Leakage   | UseExceptionHandler: מחזיר "Internal server error." בלבד
  Cross-Origin Attacks  | CORS Whitelist: 4 מקורות; AllowedHosts: localhost;127.0.0.1
  ניחוש LinkToken      | CSPRNG — RandomNumberGenerator.Fill() — 2¹²⁸ אפשרויות


21.2 תיאור ההצפנות

  מנגנון                   | תיאור
  ----------------------|----------------------------------------------------------
  גיבוב סיסמאות         | HMAC-SHA256 + Salt אקראי ייחודי לכל מנהל (PasswordHasher.cs)
  JWT                   | חתום HMAC-SHA256; מפתח מינ. 32 תווים; תוקף שעתיים
  אחסון טוקן בלקוח    | sessionStorage בלבד (לא localStorage) — נמחק עם סגירת הלשונית

14.1 פירוט API
------------------------------------------------------------------------
המערכת חושפת ממשק REST API מלא (מפותח ב-ASP.NET Core) המשמש לתקשורת מול צד הלקוח:
• AuthController: ניהול אימות (Login) והנפקת טוקנים (JWT).
• BoxController / ContainerController: ניהול נתוני ארגזים ומכולות (CRUD מלא).
• PackingJobController: ניהול משימות שיבוץ, הרצת האלגוריתמים (Run) וקבלת תוצאות (Placements).
• SessionController: ניהול סשנים שיתופיים, אישור בקשות הצטרפות (Requests) ועמדות קטלוג.
• SnapshotController: שמירה ואחזור של תמונות מצב של שיבוצים להשוואה.
• VisualizationController: אספקת נתונים מותאמים לתצוגת התלת-ממד (Three.js).

14.2 סביבת עבודה
------------------------------------------------------------------------
המערכת פותחה בארכיטקטורת Client-Server תחת סביבות הפיתוח הבאות:
• צד שרת (Backend): סביבת Visual Studio או VS Code, תחת פלטפורמת .NET 10.
• בסיס נתונים: SQL Server LocalDB, המנוהל באמצעות Entity Framework Core או חיבור ישיר (ADO.NET), כולל קובצי הגירה (SQL Scripts).
• צד לקוח (Frontend): סביבת פיתוח Web סטנדרטית (עורך קוד), מוגשת זמנית באמצעות קבצי סטטיים (wwwroot) בשרת.

14.3 שפות תכנות וטכנולוגיות
------------------------------------------------------------------------
• C# / .NET 10: שפת צד השרת ליישום ה-API, הלוגיקה העסקית והאלגוריתמים המורכבים (Greedy, Branch & Bound, Heuristic).
• SQL: שאילתות וניהול טבלאות מול מסד הנתונים OptiLoadDB.
• HTML5 / CSS3 / JavaScript (ES6+): פיתוח ממשק המשתמש הקדמי (6 דפי המערכת).
• Three.js: ספריית JavaScript לרינדור והצגה ויזואלית אינטראקטיבית של המכולה והארגזים בתלת-ממד.

הסכמה הכללית של מסד הנתונים במערכת בנויה באופן המפריד בין ישויות הבסיס, ישויות 
משימות השיבוץ, וישויות ניהול הסשנים והמשתמשים. מסד הנתונים OptiLoadDB הוא מסד יחסי

המערכת לא מבצעת שימוש בפרוצדורות מאוחסנות (Stored Procedures) או תצוגות (Views)

הערה: תרשימי ה-ERD וה-DSD הנלווים לפרק זה סופקו כקובץ תמונה/HTML נפרד

22.1 פירוט טבלאות בסיס הנתונים
------------------------------------------------------------------------

1. טבלת Admin (מנהלי מערכת)
טבלה המנהלת את מנהלי המערכת שיש להם הרשאה לבצע שיבוצים ולפתוח סשנים.
- Id: שלם (INT), מפתח ראשי (PK). חובה. מספור אוטומטי.
- Username: מחרוזת (NVARCHAR), חובה. שם משתמש, ייחודי (UNIQUE).
- PasswordHash: מחרוזת (NVARCHAR), חובה. גיבוב של הסיסמה.
- PasswordSalt: מחרוזת (NVARCHAR), חובה. מפתח ממליח לסיסמה.
- CreatedAt: תאריך (DATETIME2), חובה. זמן יצירת החשבון.

2. טבלת ContainerTemplate (תבניות מכולה)
מידות מכולות גנריות לשימוש חוזר.
- TemplateId: שלם (INT), מפתח ראשי (PK). חובה.
- TemplateName: מחרוזת (NVARCHAR), חובה. שם המכולה (למשל 'מכולה 20 רגל').
- Width, Height, Depth: ממשי (FLOAT), חובה. מידות פנימיות.
- MaxWeightKg: ממשי (FLOAT), חובה. משקל נשיאה מקסימלי.
- CreatedAt: תאריך (DATETIME2), חובה. זמן יצירה.

3. טבלת Container (אינסטנס של מכולה)
מכולה ספציפית לשימוש (הנגזרת מתבנית).
- ContainerId: שלם (INT), מפתח ראשי (PK). חובה.
- TemplateId: שלם (INT), מפתח זר (FK). חובה. קישור לתבנית אב.
- ContainerCode: מחרוזת (NVARCHAR), חובה. מספר או קוד זיהוי מכולה.
- Status: מחרוזת (NVARCHAR), חובה. זמינות המכולה.
- CreatedAt: תאריך, חובה.

4. טבלת Box (ארגזים/מוצרים)
מידות המוצרים הקיימים במערכת לפני העמסה.
- BoxId: שלם (INT), מפתח ראשי (PK). חובה.
- BoxName: מחרוזת (NVARCHAR), חובה. שם או קוד מוצר.
- Width, Height, Depth: ממשי (FLOAT), חובה. מידות פיזיות.
- WeightKg: ממשי (FLOAT), חובה. משקל ברוטו.
- IsFragile: בוליאני (BIT), חובה. האם שביר (מועמס רק למעלה מעל כולם).
- AllowRotation: בוליאני (BIT), חובה. האם ניתן לסובב את הארגז במרחב ה-3D.
- CreatedAt: תאריך (DATETIME2), חובה.

5. טבלת PackingJob (משימות העמסה ושיבוץ)
כל ריצה של האלגוריתם מתועדת כג'וב יחד עם התוצאות הסטטיסטיות שלו.
- JobId: שלם (INT), מפתח ראשי (PK). חובה.
- ContainerId: שלם (INT), מפתח זר (FK). חובה. מקושר למכולה עליה אנו עובדים.
- AdminId: שלם (INT), מפתח זר (FK). חובה. מזהה מנהל ההרצה.
- Status: מחרוזת (NVARCHAR), חובה. מצב הריצה (Completed/Failed).
- BinsUsed: שלם (INT), רשות (NULL). כמות המכולות/מיכלים שנצרכה.
- VolumeUtilization: ממשי (FLOAT), רשות (NULL). אחוז ניצולת נפחית.
- TotalWeightKg: ממשי (FLOAT), רשות (NULL). המשקל הכולל ששובץ.
- SolveTimeSeconds: ממשי (FLOAT), רשות (NULL). זמן ריצת האלגוריתם בשניות.
- IsOptimal: בוליאני (BIT), רשות (NULL). מנטר האם הפתרון נמצא אופטימלי (Branch & Bound).
- StatusMessage: מחרוזת, רשות (NULL). תיעוד שגיאה אם קרתה בריצה.
- CreatedAt, CompletedAt: תאריכים (DATETIME2). זמן התחלה וסיום.

6. טבלת PackingJobBox (ארגזים מבוקשים בג'וב)
רשימת הדרישה: כמות מכל סוג ארגז שהוגדרה להרצה בג'וב.
- JobBoxId: שלם (INT), מפתח ראשי (PK). חובה.
- JobId: שלם (INT), מפתח זר (FK). חובה.
- BoxId: שלם (INT), מפתח זר (FK). חובה.
- Quantity: שלם (INT), חובה. הכמות הנדרשת להעמסה מסוג זה.

7. טבלת PlacementResult (תוצאות המיקום המרחבי)
שומרת פרטנית את ציר ה-(x,y,z) לכל תיבה בודדת שמוקמה בתוך המכולה.
- PlacementId: שלם (INT), מפתח ראשי (PK). חובה.
- JobId: שלם (INT), מפתח זר (FK). חובה. להרצה שהפיקה תוצאה זו.
- BoxId: שלם (INT), מפתח זר (FK). חובה. איזה סוג ארגז שובץ כאן.
- InstanceIndex: שלם (INT), חובה. עבור מספר ארגזים זהים.
- BinIndex: שלם (INT), חובה. לאיזו מכולה שויך (אם קיימות מכולות רבות).
- PosX, PosY, PosZ: ממשי (FLOAT), חובה. נ"צ מרחבי 3D לעוגן (Corner).
- PlacedWidth, PlacedHeight, PlacedDepth: ממשי (FLOAT), חובה. המידות במרחב לאחר כל סיבוב (Rotation).
- RotationIndex: שלם (INT), חובה. זיהוי אופרציית הסיבוב מתוך 6 האופציות האפשריות.
- CreatedAt: תאריך, חובה.

8. טבלת ContainerSnapshot (תיעוד ויזואלי היסטורי במערכת)
שמירת תמונה (Base 64) מה-3D על מנת לקטלג היסטוריה השוואתית.
- Id: שלם (INT), מפתח ראשי (PK). חובה.
- JobId: שלם (INT), מפתח זר (FK). חובה. לאיזה ריצה (Job).
- LoadingStep: שלם (INT), חובה. שלב התיעוד בתהליך.
- BoxName: מחרוזת (NVARCHAR), חובה. תיוג.
- ImageData: מחרוזת טקסט ובינארי ארוכה מאד (VARCHAR MAX). חובה. התמונה עצמה.
- CreatedAt: תאריך (DATETIME2), רשות (NULL).

9. טבלת PackingSession (סשן עסקי רב-משתתפים בממשק המערכת)
ניהול מנגנון קבלת החלטות לייב, עריכות בסל המוצרים במקביל ע"י נציגים למפעלי יבוא/יצוא.
- SessionId: שלם (INT), מפתח ראשי (PK). חובה.
- AdminId: שלם (INT), מפתח זר (FK). חובה. יוזם ובעל הסשן.
- Name, Description: מחרוזת (NVARCHAR), חובה (שם) ורשות (תיאור).
- LinkToken: מחרוזת באורך קבוע (CHAR 36), חובה. טוקן זיהוי ייחודי להזמנות משתמשי קצה (GUID).
- Status: מחרוזת, חובה. הסשן הפעיל/סגור.
- CreatedAt: תאריך, חובה.

10. טבלת SessionUser (משתמשים זמניים המחוברים לסשנים)
עובדים מרחוק המוזמנים לסשן ופועלים בסביבתו ללא הרשמה מקדימה.
- SessionUserId: שלם (INT), מפתח ראשי (PK). חובה.
- SessionId: שלם (INT), מפתח זר (FK). חובה.
- DisplayName: מחרוזת (NVARCHAR), חובה. הכינוי הציבורי שדרשו בדיאלוג.
- Email: מחרוזת (NVARCHAR), רשות (NULL).
- Token: מחרוזת (CHAR 36), חובה. זיהוי אסימון הרשאות המצומד לסשן. ייחודי.
- RootToken: מחרוזת קבועה, חובה. כלי מעקב.
- CreatedAt: תאריך, חובה.

11. טבלת AccessRequest (אישורי כניסה לחדרי סשנים אונליין)
- RequestId: שלם (INT), מפתח ראשי (PK). חובה.
- SessionId, SessionUserId: מפתחות זרים של סשן ומשתמש. חובה.
- Status: מחרוזת (NVARCHAR), חובה. יכול להיות: Pending (ממתין), Approved (מאושר), Denied.
- RequestedAt: תאריך, חובה.
- RespondedAt: תאריך, רשות. זמן המענה לקבלת הבקשה.

12. טבלת SessionBox (ארגזים בסשן זמני - הסל)
הרכיב המאפשר למספר נציגים לבנות ביחד סל ארגזים עתידי להרצת אופטימיזציה בהשהייה.
- SessionBoxId: שלם (INT), מפתח ראשי (PK). חובה.
- SessionId: שלם (INT), מפתח זר (FK). חובה.
- BoxId: שלם (INT), מפתח זר (FK). חובה.
- Quantity: שלם (INT), חובה. כמה פריטים התבקשו.
- AddedBy: מחרוזת, חובה. איזה נציג הוסיף את הכמות.
- AddedAt: תאריך, חובה.

13. טבלת BoxAuditLog (יומן שינויים ושליטה בסשן)
יצירת עקבות ביקורת (Audit Trails) למניעת מחיקות ועריכות ארגזים לא שקופות בין הלקוחות במערכת.
- LogId: שלם (INT), מפתח ראשי (PK). חובה.
- SessionId: שלם (INT), מפתח זר (FK). חובה.
- Action: מחרוזת (NVARCHAR), חובה. סוג הפעולה כגון: Addition, Deletion.
- BoxId: שלם (INT), רשות. רלוונטי במקרה של מחיקות ארגז.
- BoxName: מחרוזת (NVARCHAR), חובה. נתון ארכיוני.
- Quantity: שלם, רשות.
- ChangedBy, ChangedByType: מחרוזת, חובה. שם המבצע והתפקיד שלו (Admin / User).
- ChangedAt: תאריך, חובה.

14. טבלת ErrorLog (ניטור כשלים ואלגוריתמיקה)
מנגנון תחזוקתי לתחקור באגים, קריסות מנוע, והליכי הרצת אלגוריתמי 3D שונים.
- ErrorId: שלם (INT), מפתח ראשי (PK). חובה.
- JobId: שלם (INT), מפתח זר (FK), רשות (NULL) מתעד באג שארע בקונטקסט של ג'וב במידה וקיים.
- Context, Message, StackTrace: מחרוזות (NVARCHAR), חובה ורשות. דיווח נפילה, התראות קוד, מעקב.
- CreatedAt: תאריך, חובה.
